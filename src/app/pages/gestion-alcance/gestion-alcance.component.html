Gestión del Alcance del Proyecto

Transformación Digital — Florería Adonai

Fase 3 — Planificación

La Florería Adonai es una microempresa local que opera principalmente de forma presencial y por WhatsApp. Su gestión operativa y comercial presenta oportunidades de mejora: el inventario se controla visualmente, las ventas se registran en un cuaderno físico, los clientes frecuentes no están organizados y la presencia digital del negocio es limitada. Este documento desarrolla la Gestión del Alcance del Proyecto que busca transformar digitalmente esa operación mediante un sistema web propio en Django.

Técnica 3

Análisis de Alternativas

Evaluación de tres enfoques metodológicos de alcance para determinar cuál se adapta mejor a una microempresa sin sobrecargarla administrativamente.

Evaluación de Enfoques de Alcance

Opción A
Gestión Ágil Ligera
  • ✓ Aplicación inmediata y de bajo costo.
  • ✓ Muy flexible ante cambios imprevistos.
  • ✗ Sin trazabilidad formal de decisiones y requisitos.
  • ✗ Dificulta el control y la aceptación formal del cliente.
✗ Descartada
Opción B
PMBOK Tradicional
  • ✓ Rigor absoluto en entregables y trazabilidad.
  • ✓ Planificación exhaustiva y control formal de cambios.
  • ✗ Exceso de documentación para una microempresa.
  • ✗ Requiere alta carga de tiempo administrativo de la dueña.
✗ Descartada
Opción C
Gestión Mixta Adaptada
  • ✓ Conserva rigor estructurado del PMBOK (Entradas, Herramientas, Salidas).
  • ✓ Simplifica la documentación al tamaño real de la florería.
  • ✓ Asegura trazabilidad sin burocracia excesiva.
  • ✓ Fácil adopción por la propietaria y el equipo técnico.
✓ Seleccionada
Justificación de la Selección: La alternativa mixta adaptada permite mantener el rigor formal (entregables trazables y controlados) sin abrumar a las usuarias operativas del negocio, logrando un balance óptimo entre formalidad de TI y agilidad práctica.

2.3 Salidas (Clasificación de Requisitos)

Requisitos de Negocio

ID Requisito de negocio
RN-01 Mejorar el control del inventario para reducir la dependencia de conteos visuales y manuales.
RN-02 Centralizar el registro de ventas para contar con información confiable y accesible del negocio.
RN-03 Organizar la información de clientes frecuentes para fortalecer la atención personalizada.
RN-04 Aumentar la presencia digital de la Florería Adonai como canal de atención y promoción.

Requisitos Funcionales

ID Categoría Descripción del Requisito Causa que atiende
RF-01 Inventario El módulo de inventario del sistema debe permitir registrar productos clasificados por categoría (flores frescas, flores secas, complementos, temporada). Control informal del inventario.
RF-02 Inventario El módulo de inventario del sistema debe permitir registrar entradas y salidas de productos, y mostrar alertas automáticas de stock mínimo. Control informal del inventario.
RF-03 Ventas El módulo de ventas del sistema debe permitir registrar cada venta con: fecha, producto, cantidad, precio unitario y canal de venta (presencial o WhatsApp). Registro manual e incompleto de ventas.
RF-04 Clientes El módulo de clientes del sistema debe permitir registrar nombre, número de contacto, preferencias de compra e historial básico de pedidos. Gestión desordenada de clientes.
RF-05 Presencia Digital El catálogo digital integrado debe mostrar los productos con fotos reales, precios y permitir un flujo de compra simulada (envío de pedido por WhatsApp o Yape). Baja presencia digital de la florería.
RF-06 Atención Comercial El canal de WhatsApp Business debe centralizar la atención de pedidos de forma separada de la comunicación personal de la propietaria. Atención comercial poco organizada y dependiente.
RF-07 Panel Administrativo El panel interno del sistema debe contar con control de acceso por roles (staff/superuser) para proteger la información del negocio. Falta de un sistema seguro y centralizado para la administración.

Requisitos No Funcionales

ID Requisito No Funcional Justificación del Requisito
RNF-01 El sistema web no debe generar costo de licenciamiento para el negocio (desarrollo propio). La Florería Adonai es una microempresa sin presupuesto para licenciar software comercial de terceros.
RNF-02 Las herramientas deben poder utilizarse desde un teléfono celular. La propietaria y la vendedora gestionan el negocio principalmente desde sus dispositivos móviles en tienda.
RNF-03 El panel interno debe requerir solo una capacitación breve y no conocimientos de programación. Ninguna de las usuarias operativas del negocio posee formación técnica en sistemas de computación.
RNF-04 La atención comercial digital debe mantenerse separada del WhatsApp personal. Evita la desorganización de pedidos y permite establecer un horario comercial claro.
RNF-05 El sistema debe contar con rutas internas protegidas (seguridad). Garantiza que usuarios no autorizados no accedan a los datos financieros de la tienda.
RNF-06 El código del sistema debe ser mantenible y sostenible a futuro. Permite agregar nuevas características o escalar el proyecto posteriormente sin reconstruirlo desde cero.

3.3 Salidas (Declaración del Alcance del Proyecto)

Descripción del Proyecto

Desarrollar y desplegar un sistema web propio en Django para la Florería Adonai que mejore la gestión operativa y comercial, orientándose a organizar el inventario, centralizar las ventas, gestionar clientes frecuentes y ofrecer un catálogo digital integrado.

Alcance del Producto

El producto consiste en un sistema web compuesto por: módulo de inventario con alertas de stock, módulo de ventas por canal, módulo de gestión de clientes, un catálogo público con flujo de compra simulada (vía WhatsApp/Yape), un panel administrativo protegido con roles y la integración a WhatsApp Business para la recepción de pedidos, junto a una guía de uso para la propietaria.

Entregables Principales y Criterios

# Entregable Criterio de Aceptación
1 Módulo de inventario del sistema Mínimo 4 categorías activas, campos de entrada/salida y alertas automáticas de stock mínimo funcionando.
2 Módulo de registro de ventas Formulario web con campos de fecha, producto, cantidad, precio y origen del canal de venta.
3 Módulo base de clientes Estructura de registro con datos de contacto, tipo de flor preferida e historial de compras.
4 Catálogo público con compra simulada Mínimo 10 arreglos florales y productos listados con fotografía real, precio y botón funcional de pedido por WhatsApp.
5 Panel administrativo con control de acceso Rutas internas protegidas con HTTP 403 para visitantes no autorizados, dashboard funcional y diferenciación de roles (staff/superuser).
6 Configuración de WhatsApp Business Perfil comercial activo con etiquetas de pedidos, conectado correctamente con el flujo del catálogo público.
7 Guía básica de uso Guía aprobada e impartida para manejar el panel interno y catálogo de forma fluida.
Exclusiones Explícitas
  • Sistema ERP empresarial.
  • Aplicación móvil nativa.
  • Pasarela de pago electrónica real.
  • Facturación SUNAT.
  • Costo de internet o equipos.
Restricciones Clave
  • Sin costo de licenciamiento para el negocio.
  • Debe poder usarse en celulares.
  • No requiere código avanzado (para el negocio).
Supuestos Aceptados
  • Conectividad a internet estable en el local.
  • Participación activa de la dueña.
  • Operación continua en tienda.
  • Fotos reales de arreglos florales.
  • Celular con acceso a internet.

1.1.1.1 Estructura de inventario por categorías Desarrollo del modelo de datos e interfaz en Django para gestionar productos por categorías, con campos de entrada y salida de stock. Equipo del proyecto Mínimo 4 categorías activas y CRUD de productos funcional en el sistema.
1.1.1.2 Alertas de stock mínimo Programación de alertas automáticas en el sistema web para identificar productos con stock bajo en el panel administrativo. Equipo del proyecto Alerta visible en el dashboard ante stock bajo; validada por la propietaria.
1.1.2.1 Registro de productos reales Carga inicial de datos de inventario del negocio (precios y cantidades de inicio) en la base de datos del sistema web. Equipo del proyecto Al menos 20 productos registrados con su stock inicial real.
1.2.1.1 Campos mínimos de venta Desarrollo del formulario web y modelo de base de datos para el registro de ventas diarias. Equipo del proyecto Formulario web con los 5 campos mínimos de control funcionando.
1.2.2.1 Vista diferenciada por canal Dashboard o filtros en el panel administrativo para visualizar ventas y consolidar estadísticas por canal. Equipo del proyecto Vista que agrupe ventas presenciales y por WhatsApp de forma diferenciada.
1.3.1.1 Campos de clientes Desarrollo del modelo de clientes y tabla interactiva para capturar y ordenar datos de contacto e historial. Equipo del proyecto Gestión de clientes funcional en el sistema, permitiendo búsqueda y registro.
1.3.2.1 Registro de contactos habituales Migración inicial de al menos 15 clientes habituales de WhatsApp y del cuaderno físico a la base de datos. Equipo del proyecto 15 registros iniciales reales de clientes cargados en el sistema.
1.4.1.1 Fotografías de productos Captura y edición básica de fotografías reales, subidas y gestionadas a través del sistema web (media files). Equipo del proyecto Al menos 10 fotografías reales cargadas y mostradas en el catálogo.
1.4.2.1 Diseño del catálogo público Desarrollo de la vista pública del catálogo (front-end) integrada al backend de Django con productos, fotos y precios. Equipo del proyecto Catálogo web navegable, responsivo, listando productos activos.
1.4.3.1 Flujo de compra simulada Desarrollo del carrito de compras web con redirección a WhatsApp para finalizar la venta y pasarela simulada por Yape. Equipo del proyecto Botón de pedido funcional que genera un mensaje prellenado en WhatsApp.
1.5.1.1 Datos del perfil del negocio Configuración del perfil comercial en WhatsApp Business (horarios, dirección, descripción). Equipo del proyecto Perfil verificado comercialmente en el celular de la propietaria.
1.5.1.2 Etiquetas de pedidos Creación de etiquetas ("Pendiente de pago", "Por entregar", "Pagado") para el control visual del chat. Equipo del proyecto Mínimo 4 etiquetas configuradas y activas.
1.5.1.3 Respuestas rápidas Redacción y carga de mensajes automatizados (precios base, métodos de pago, ubicación). Equipo del proyecto Mínimo 5 mensajes rápidos configurados en el celular.
1.6.1.1 Material de capacitación Elaboración de un manual sencillo paso a paso con imágenes explicativas para el uso diario del sistema web (catálogo, panel interno) y WhatsApp Business. Equipo del proyecto Guía interactiva en formato digital o folleto, validada por el docente.
1.6.2.1 Sesión de presentación y demo Sesión de capacitación práctica donde se simula el registro de ventas, inventario y atención comercial en el sistema. Equipo del proyecto Firma de conformidad y capturas de pantalla de la propietaria operando el sistema.
1.7.1.1 Control de acceso y rutas protegidas Implementación de autenticación de usuarios y protección de rutas administrativas (decorators de Django). Equipo del proyecto Acceso restringido al panel interno; redirección a login para usuarios no autorizados.
1.7.2.1 Dashboard de KPIs Desarrollo de un panel de métricas con indicadores clave (ventas del día, productos más vendidos). Equipo del proyecto Dashboard funcional mostrando métricas actualizadas de la base de datos.
1.8.1.1 Despliegue en entorno de producción Configuración de servidor VPS/PaaS (ej. Render/Railway), base de datos de producción (PostgreSQL) y dominio. Equipo del proyecto Sistema web accesible públicamente a través de internet sin errores 500.
1.8.2.1 Suite de pruebas automatizadas Creación de pruebas unitarias y de integración para validar el flujo del inventario y las ventas. Equipo del proyecto Tests pasando correctamente en el entorno de desarrollo y CI/CD.
